## Context

The viewer HTML is generated by `src/renderer/html-builder.js`. It already has a tab model:

- `tabs` — array built by `extractTabs(ALL_FLOWS)`, one entry per Node-RED `tab` node, rebuilt from scratch every time `ALL_FLOWS` changes (WebSocket push or manual refresh).
- `buildTabData(flows, tabId, dirtySet)` — generic: it filters nodes by `n.z === tabId` and works for **any** id, including a subflow definition id, because a subflow's child nodes also use `z` to point at their owning subflow. No change needed here.
- `switchTab(tabId)` — renders whichever tab is requested and remembers zoom/pan per tab in `zoomState`.
- Nodes only have `mouseenter`/`mouseleave` handlers (tooltip). There is no click/dblclick handler today — confirmed intentional, out of scope for this change.
- Subflow instance nodes are already visually marked with an "S" badge and colored `#9BC7D4` (see `getNodeColor`).

The gap: subflow definitions are never rendered as tabs, and there is no way to navigate from a subflow instance to its definition.

## Goals / Non-Goals

**Goals:**
- Double-click on a subflow instance node (`type` starts with `subflow:`) opens its subflow definition as a tab.
- Re-opening an already-open subflow tab activates it in place — no duplicate, no repositioning.
- New subflow tabs are inserted immediately after the tab that was active at open time.
- Subflow tabs are visually distinguished by a close ("×") control; regular flow tabs are not closable.
- Subflow tabs render with the exact same pipeline as regular tabs (`buildTabData` + `render`), so node/link/group/dirty-highlight behavior is identical.
- Subflow tabs that reference a deleted subflow definition are dropped automatically on the next flow update.

**Non-Goals:**
- Opening node "detail" panels/editors — explicitly out of scope, current no-op behavior is correct and unchanged.
- Nested position bookkeeping across a subflow tab that is itself later re-anchored (e.g., if the tab it was opened "after" is closed, the subflow tab simply keeps its current array position — no re-anchoring logic).
- Preserving subflow tab open state across a full page reload (client-only, in-memory state).
- Reordering regular flow tabs relative to each other when new ones appear from a live flow update (existing append-only behavior for brand-new tab nodes is out of scope; rare case, addressed as a known limitation below).

## Decisions

### Decision 1: Represent subflow tabs as entries in the same `tabs` array, tagged with `kind`

**Rationale:** `tabs` already drives `buildTabBar()`, `switchTab()`, and `zoomState`. Reusing the same array (each entry gets `kind: 'tab'` or `kind: 'subflow'`) means the render/switch code needs one small branch (for the close button) instead of a parallel tab model with its own merge logic.

**Alternative considered:** A separate `subflowTabs` array merged with `tabs` at render time. Rejected — every consumer (`switchTab`, `buildTabBar`, dirty-indicator refresh) would need to know about two arrays and re-derive the merged order, doubling the surface for bugs versus one array with a discriminator field.

### Decision 2: Stop wholesale-replacing `tabs` on flow updates; reconcile instead

**Rationale:** Today, `tabs = extractTabs(ALL_FLOWS)` on every WebSocket message and refresh wipes out any subflow entries. The fix is a `reconcileTabs(freshRegularTabs)` step that:
1. Drops `kind: 'subflow'` entries whose subflow definition no longer exists in `ALL_FLOWS`.
2. Updates in place the name/disabled metadata of existing `kind: 'tab'` entries that are still present.
3. Drops `kind: 'tab'` entries for tab nodes that were deleted.
4. Appends any brand-new regular tab nodes (present in `freshRegularTabs` but not yet in `tabs`) at the end.

This keeps existing subflow tabs and their positions intact across live updates, only touching entries that actually changed.

**Alternative considered:** Snapshot-and-restore subflow tabs after each wholesale rebuild (save subflow entries, rebuild `tabs`, splice them back at their old index). Rejected — index-based splice-back breaks as soon as regular tabs are added/removed/reordered, silently landing a subflow tab in the wrong place; reconciling by id is more robust and no more code.

### Decision 3: Anchor insertion position at open time, not by ongoing tracking

**Rationale:** When a subflow tab is opened, compute `activeIndex = tabs.findIndex(t => t.id === currentTabId)` and `tabs.splice(activeIndex + 1, 0, newEntry)` once. No persistent "anchor id" is stored afterward — the tab simply lives at that array position like any other entry, and subsequent opens/closes work with plain array indices. This matches the required behavior ("opens to the right of the active tab" and "does not move once opened") with no extra bookkeeping.

**Alternative considered:** Store `afterId` on each subflow tab and re-resolve its position relative to its anchor on every render. Rejected — unnecessary complexity; the proposal only requires correct placement at open time and stability afterward, not dynamic re-anchoring if the anchor tab is later closed or moved.

### Decision 4: Tab bar visibility becomes dynamic (`tabs.length > 1`) instead of a fixed `hasTabs` flag

**Rationale:** Today `hasTabs` is computed once from the initial flow data; a single-tab flow (no explicit `tab` nodes) never shows a tab bar. Once a subflow tab can be opened in that scenario, the bar must appear so the user can see and close it. `buildTabBar()` now decides visibility from the live `tabs.length > 1` check on every call instead of relying on the stale flag computed at load/refresh time.

**Alternative considered:** Force `hasTabs = true` permanently once any subflow tab is opened and never revert. Rejected — after closing the last subflow tab the bar should disappear again if there was only ever one implicit flow tab; recomputing on every `buildTabBar()` call handles both directions for free.

### Decision 5: Double-click handler gated by `type.indexOf('subflow:') === 0`

**Rationale:** Matches the existing check already used for the "S" badge and node color (`getNodeColor`, badge-rendering block), so subflow detection stays consistent with what's already visually marked as a subflow instance. The subflow definition id is `d.type.slice('subflow:'.length)`.

**Alternative considered:** Add a `subflowId` field to each node's data object in `buildTabData` (looked up once) instead of parsing `type` again. Rejected — the definition id is a pure string slice already computed inline elsewhere (`getSubflowInstanceName`); adding a field for a one-line derivation is unnecessary.

### Decision 6: No parent/child relationship is stored between subflow tabs

**Rationale:** The dblclick handler and `openSubflowTab`/`closeSubflowTab` are attached uniformly to every rendered tab, regardless of whether it's a regular flow tab or a subflow tab. Because a subflow's own child nodes can themselves be subflow instances, double-clicking one while its parent subflow tab is active opens a new tab the same way, anchored after the currently active (parent) tab — nesting is handled for free by the existing generic logic, with no depth limit and no special-casing. Each `tabs` entry is independent: there is no `parentId` field and `closeSubflowTab` only ever removes the single entry it's called with, so closing a subflow tab never cascades to tabs opened from within it.

**Alternative considered:** Track a `parentId` per subflow tab and cascade-close descendants when a parent tab is closed. Rejected — the proposal explicitly requires the opposite behavior (closing a parent must NOT close its children), and tracking parentage would add state with no required use.

## Risks / Trade-offs

- **Appending brand-new regular flow tabs at the end (Decision 2, step 4) instead of at their authored position** → Acceptable; regular tabs added while a subflow tab is open is a rare live-editing scenario, and the tab still appears and works correctly, just possibly out of authored order until the next full reload.
- **No persistence of open subflow tabs across page reload** → Acceptable per Non-Goals; this is a lightweight, session-local viewer feature, not a saved workspace state.
- **Deleting a subflow definition while its tab is the active tab** closes it and falls back to an adjacent tab → Mitigation: reuse the same fallback logic as Decision 2/close-button handling (nearest remaining tab, or empty state if none left).
