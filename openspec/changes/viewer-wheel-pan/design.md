## Context

The flow viewer (generated by `src/renderer/html-builder.js`) embeds a D3.js-based SVG viewer in the HTML output. Currently it uses `d3.zoom()` with default settings, which maps mouse wheel to zoom and ignores wheel tilt events. The Node-RED editor itself uses wheel for vertical pan and wheel tilt for horizontal pan. This design aligns the viewer's interaction model with the editor.

Current implementation at `src/renderer/html-builder.js` around line 374:

```js
var zoom = d3.zoom().scaleExtent([0.1, 2]).on("zoom", function(event) {
    contentGroup.attr("transform", event.transform);
    if (currentTabId) zoomState[currentTabId] = event.transform;
});
svg.call(zoom);
```

## Goals / Non-Goals

**Goals:**
- Wheel up/down pans the view vertically
- Wheel tilt left/right pans the view horizontally
- Pinch-to-zoom and drag-to-pan keep working exactly as before
- Pan operations update `zoomState` so per-tab position is preserved

**Non-Goals:**
- Changing zoom level range or scale extent
- Adding zoom UI controls (buttons, sliders)
- Modifying the Mermaid viewer or any non-SVG viewer

## Decisions

### Decision 1: Filter out wheel events from d3.zoom, handle wheel manually

**Rationale:** d3.zoom's built-in wheel handling does zoom. Disabling it via `zoom.filter()` while adding a separate native `wheel` event listener gives full control over pan behavior without losing pinch-to-zoom (which goes through d3.zoom's touch handling).

**Alternative considered:** Using `zoom.wheelDelta()` to invert the behavior. Rejected — d3.zoom wheel handling always produces a scale+translate transform, not a pure translate. The math to force pure pan through the zoom transform is fragile and hard to read.

### Decision 2: Use native `wheel` event with `deltaY` and `deltaX`

**Rationale:** The `wheel` event provides `deltaY` for vertical scroll and `deltaX` for horizontal tilt in a single handler. Compute new translate values and apply via `d3.zoomIdentity.translate().scale()` so the transform stays in sync with `zoomState`.

**Alternative considered:** Separate `wheel` and `wheel` with axis detection. Rejected — `deltaX`/`deltaY` on the single event already distinguishes vertical from horizontal.

### Decision 3: Apply pan speed factor of 1.0 (no multiplier)

**Rationale:** Match Node-RED editor behavior where 1 wheel tick = 1 pixel pan at current zoom level. Keep it simple; users can adjust later if needed.

## Risks / Trade-offs

- **Touchpad two-finger scroll will also pan** instead of zoom. This matches Node-RED editor behavior but differs from the previous viewer. Users who relied on touchpad-scroll-to-zoom will need to use pinch instead. → Acceptable trade-off; pinch is the standard zoom gesture.
- **No configuration option for pan speed.** → Low risk; can add later if users request it.
